The codebase is mid-migration: a 9,218-line monolith (script.js) plus a 3,146-line god object (planeObject.js), with a small bridgehead of ESM modules in src/ (PlaneModel, PlaneRenderer, normalizeAircraft). The new modules are thin and shallow; the real depth and logic remain in the legacy files. The seam between old and new is not yet committed to, creating two sources of truth for almost every core concept.

Candidate 1 — AircraftUpdatePipeline

A deep module that owns the complete ingest cycle: wire data → normalized model → state delta

Strong
Files involved
html/js/script.js (processAircraft, fetchDone) src/normalizeAircraft.js src/PlaneModel.js html/js/planeObject.js (updateData calls)
Problem

The aircraft update pipeline is not a module at all — it is a sequence of function calls spread across three files with global side-effects at each step. fetchDone() unpacks the JSON, processAircraft() normalizes the wire format (old code), normalizeAircraft() normalizes it again (new code), and then PlaneModel.updateData() is called from inside the old update path. The seam between "data arrived" and "model is up to date" is wide open: tests cannot exercise this pipeline without a full browser environment, because every step mutates globals (g.planes, SelPlanes) and reaches into the DOM.

The interface is as complex as the implementation: to call the pipeline, you must know the exact shape of the global state, the DOM elements that hold configuration, and the order in which functions must fire. There is no locality — the same concept (normalize this aircraft) exists in two places simultaneously.

Solution

Extract a single deep AircraftUpdatePipeline module. Its interface is one method: pipeline.ingest(wireFrame) which accepts the raw aircraft.json payload and returns an array of AircraftStateChange objects — plain data describing what changed. The pipeline owns all normalization, model updates, and staleness evaluation. The rest of the app reacts to state-change events rather than being called directly.

The key move: accept a PlaneStore dependency (an object that holds aircraft by ICAO) rather than reading from g.planes. The store can be an in-memory fake in tests, Postgres-backed in a hypothetical server mode, or the existing global in production — but it is the pipeline's only dependency, not a side-effect.

Benefits — Locality & Leverage
  • Locality: All normalization, validation, and staleness logic in one place. Change the wire format → edit one file.
  • Leverage: Every feature that needs to react to aircraft updates (rendering, table, history) calls ingest() — no feature needs to know about JSON structure.
  • Testability: Tests inject a plain wire frame, assert on returned state-change objects. No DOM, no OpenLayers, no global state. The old 99%-untested pipeline becomes fully unit-testable.
  • Seam clarity: The seam between "data" and "display" is now real and explicit. Rendering can subscribe to state-change events rather than being called imperatively.

Candidate 2 — Commit the PlaneModel Seam (eliminate PlaneObject)

The in-progress migration created two sources of truth; the seam must be committed to or abandoned

Strong
Files involved
src/PlaneModel.js html/js/planeObject.js (3,146 lines) src/PlaneRenderer.js html/js/script.js (PlaneObject references)
Problem

The modernization introduced PlaneModel (49 lines, tested) as a replacement for PlaneObject (3,146 lines, untested). Both exist simultaneously. Both hold aircraft data. A position update writes to PlaneObject.lat and PlaneModel.lat. If one is updated without the other, the display shows stale data with no error.

There is no one adapter — there are two implementations of the same concept at an uncommitted seam. This is not "one adapter means hypothetical, two means real." It is two implementations with no deliberate seam at all, just migration debt.

Deletion test: if you delete PlaneModel, complexity reappears in PlaneObject (it already is there). If you delete PlaneObject, complexity must move somewhere — but it's recoverable. PlaneObject is not earning its keep as a separate structure; it's earning its keep as the only structure.

Solution

Commit the seam: make PlaneModel the one canonical aircraft data structure, and reduce PlaneObject to a rendering adapter — an object that holds an OL feature and syncs it from a PlaneModel. No aircraft data in the adapter. Volatile state (selected, visible, trail) either moves into PlaneModel or into separate stores keyed by ICAO.

PlaneRenderer already exists for this; it should be the only rendering adapter. PlaneObject's icon-caching and style logic moves into a MarkerPipeline module (Candidate 3).

Benefits — Locality & Leverage
  • Locality: One place where aircraft data lives. No risk of divergence between two models.
  • Leverage: PlaneModel's 121 lines of test coverage now covers all aircraft data — not just 49 lines of a shadow model.
  • Testability: Rendering tests only need a PlaneModel + a mock OL feature. Today they need PlaneObject, script.js globals, and a DOM.

Candidate 3 — StalenessOracle

Three files define "when is an aircraft dead" — concentrate them into one

Worth exploring
Files involved
src/PlaneModel.js (isStale — 15s/30s/120s) html/js/script.js:reaper() (removes from map) html/js/planeObject.js (track fade logic)
Problem

The concept of "aircraft staleness" has three separate implementations. PlaneModel.isStale() checks timestamps against type-based thresholds (15s for ADSB, 30s for MLAT, 120s for ADS-C). script.js:reaper() also checks timestamps and removes planes from the map on a different schedule. planeObject.js fades tracks based on yet another implicit threshold. Changing the ADS-C timeout from 120s to 180s requires editing at least two files — possibly three.

This is a shallow seam: the interface (is this aircraft stale?) is simple; the three implementations each add their own variation on the rule.

Solution

Extract a StalenessOracle module. Interface: oracle.evaluate(model, now) → StalenessLevel (fresh | stale | dead). The oracle knows the thresholds by source type and exposes them as named constants. Reaper, track fader, and icon updater all call oracle.evaluate() — they never reason about thresholds themselves.

Tests verify the thresholds once, in one place. The existing PlaneModel.isStale() can become the oracle's entry point or be absorbed into it.

Benefits — Locality & Leverage
  • Locality: Timeout change = one file, one constant. No searching three files.
  • Leverage: Three callers share one implementation. Add a new source type → update one threshold table.
  • Testability: Threshold tests already exist in PlaneModel.test.js — they just need to cover all three levels instead of one boolean.

Candidate 4 — MarkerPipeline

Icon selection is scattered across four files; concentrate it behind a single interface

Worth exploring
Files involved
html/js/markers.js (1,495 lines — SVG definitions) html/js/script.js:updateIcon() (selection logic) html/js/planeObject.js (markerStyle cache) html/js/defaults.js (colors per source type)
Problem

Answering "what icon does this aircraft display?" requires reading four files: defaults.js for color per source type, markers.js for SVG definitions keyed by aircraft category, script.js:updateIcon() for the selection logic (military? helicopter? glider?), and planeObject.js for the cache. The interface is shallow: to call updateIcon() you must know the PlaneObject, the global config, and the current filter state. Adding a new aircraft category touches all four files.

Solution

Extract a MarkerPipeline module. Interface: pipeline.select(model, config) → MarkerSpec (SVG string + color + scale). The pipeline owns the decision tree (military → helicopter → glider → default), the SVG lookup, the color resolution, and the cache. Callers receive a MarkerSpec and render it; they never reason about aircraft categories.

Benefits — Locality & Leverage
  • Locality: Adding a new aircraft category = one place. Tests confirm the right marker is selected.
  • Leverage: Table view, info panel, and map icon all get the same marker from one pipeline — no inconsistency.
  • Testability: pipeline.select() is a pure function. Tests inject a PlaneModel, assert on a MarkerSpec. No DOM, no OL canvas.

Candidate 5 — Extract formatter.js to a deep DisplayFormatter

The one module that could be tested today but isn't — low effort, immediate wins

Speculative
Files involved
html/js/formatter.js (453 lines) html/js/defaults.js (unit prefs)
Problem

formatter.js is entirely pure functions (unit conversion, precision rounding, null handling) but has zero tests. It reads unit preferences from globals rather than accepting them as arguments, so it can't be called without the global being set up. This is fixable without a large refactor.

Solution

Accept unit preferences as a parameter (dependency injection). Return display strings, don't mutate DOM. All 453 lines become testable pure functions with trivial setup. Mark as speculative because the deletion test is weak: formatter.js is already close to what it should be; this is polish, not a deep architectural change.

Top Recommendation

Start with Candidate 2 (commit the PlaneModel seam), then immediately follow with Candidate 1 (AircraftUpdatePipeline).

The two-model problem is an active desync risk: every new feature must update both PlaneObject and PlaneModel or introduce a subtle bug. Committing to PlaneModel first shrinks the target surface, so Candidate 1's pipeline has a clean model to write to. The combination removes the largest source of test debt (the pipeline is testable only after the model is single) and eliminates the most common cause of future bugs (dual-update requirement). Candidates 3 and 4 are high-value but can follow in any order once the seam is clean.
